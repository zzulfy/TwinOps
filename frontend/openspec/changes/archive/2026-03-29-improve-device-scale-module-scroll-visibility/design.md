## Context

设备规模模块目前位于固定高度的卡片面板中，内容区高度受限。当条目数量或条目高度超过可视空间时，底部条目会被遮挡，用户无法完整浏览。当前布局中存在“父容器裁切 + 子内容固定网格”的组合风险：父容器可能通过 overflow 裁切内容，而子模块未提供稳定的滚动通道。

相关约束：
- 该模块位于看板密集布局中，不能无限扩张高度影响整体栅格。
- 视觉风格需保持现有白色主题与信息层级。
- 需要在桌面与移动断点下维持可达性与可读性。

## Goals / Non-Goals

**Goals:**
- 在设备规模模块内容超出可视高度时提供稳定的模块内纵向滚动。
- 明确父子容器职责，避免“看得见滚动条但内容仍被裁切”或“无滚动通道”。
- 保持现有卡片视觉语言，避免因修复可达性引入大范围样式回归。

**Non-Goals:**
- 不重做整个 Dashboard 栅格系统。
- 不引入分页、虚拟滚动或后端数据改造。
- 不改变设备规模模块的数据口径与统计逻辑。

## Decisions

### Decision 1: 优先采用“模块内滚动”而非全局面板滚动
- 方案：在设备规模模块内部建立唯一滚动容器，内容溢出时由模块自身处理。
- 原因：局部改动可控，避免影响其他面板交互和已稳定模块。
- 备选方案：
  - 将通用面板 body 统一改为可滚动：影响面过大，易引入连锁回归。
  - 改为“更多/分页”：会增加一次交互，不满足“信息立即可见”的目标。

### Decision 2: 解除固定行数网格对内容高度的硬性约束
- 方案：保留卡片网格表达，但避免使用强绑定 3 行固定分配高度策略；允许内容自然扩展并由滚动容器承接。
- 原因：固定行高在信息密度变化时最易造成底部遮挡。
- 备选方案：继续固定行数并压缩字体/间距，不足以从根因解决可达性问题。

### Decision 3: 父容器只负责外观边界，子容器负责可达性
- 方案：若父容器存在 overflow 限制，子容器必须提供明确可滚动路径；必要时微调父容器裁切策略，避免双重裁切。
- 原因：将“外观裁切”和“内容可达”职责分离，降低布局耦合。
- 备选方案：完全去掉父容器裁切，可能破坏面板圆角与装饰层视觉约束。

### Decision 4: 滚动体验采用“低干扰可发现”策略
- 方案：滚动条样式轻量化；可选增加底部渐隐提示，提示仍有内容。
- 原因：既保证可发现性，又不破坏看板美观度。
- 备选方案：默认粗滚动条，发现性高但视觉侵入明显。

## Risks / Trade-offs

- [Risk] 不同浏览器滚动条表现不一致 → Mitigation: 采用标准属性 + 渐进增强样式，不依赖单一私有实现。
- [Risk] 父容器和子容器同时滚动导致交互混乱 → Mitigation: 明确仅保留一条主滚动路径，验证滚轮与触控行为。
- [Risk] 模块内滚动降低一次性信息总览能力 → Mitigation: 控制卡片最小尺寸与间距，保证首屏可见信息量。
- [Risk] 移动端高度更紧张导致可读性下降 → Mitigation: 在断点下同步验证条目高度、字体和可滚动可达性。
