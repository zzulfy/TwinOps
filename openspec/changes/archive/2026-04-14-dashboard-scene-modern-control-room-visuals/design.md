## Context

当前右侧仿真区已经从后端驱动 UI 转向前端持有设备目录，也已经具备室内相机约束、左键旋转/平移分区和设备详情中央弹窗。  
但这些能力更多解决了“能用”，没有真正解决“好看”：

- 房间空间比例先天偏空，地板面积过大。
- 柜列布置更像边缘摆放，缺乏近中远的视觉节奏。
- 相机经常把用户带到房间几何中心，导致通道像主角、设备像配角。
- 标题板和面板容器仍带有后台卡片气质，破坏沉浸感。

这次变更的目标不是引入新引擎或新资产，而是在当前 Three.js 程序化场景基础上，重新定义空间语言、设备构图、默认镜头和场景叠层。

约束：

- 必须保持当前 32 台演示设备与数据库设备 1:1。
- 必须保持现有点击命中、中央弹窗与状态展示链路。
- 不引入新的运行时三维资产管线。
- 不破坏现有桌面端左键边缘平移/中部旋转和室内范围约束。

## Goals / Non-Goals

**Goals**

- 让用户打开页面时先看到设备，而不是空地板或空墙。
- 将右侧仿真区重构为“现代室内控制室展示廊”，形成清晰的前、中、后、端墙层次。
- 缩窄通道视觉占比，让两侧设备平台成为主构图面。
- 让不同 `visualFamily` 在轮廓和面板语言上可以被快速区分。
- 让场景标题板和右侧容器样式与仿真空间气质统一。

**Non-Goals**

- 不把当前程序化场景替换成完整 GLB 写实资产管线。
- 不改变左侧业务模块布局。
- 不新增新的业务数据字段或后端接口。
- 不在本次变更中实现自由漫游或复杂镜头预设系统。

## Decisions

### Decision 1: 视觉重心改为“设备主导”，而不是“房间展示”

- 方案：缩窄中轴通道，建立两侧设备平台，让近景柜体靠近镜头并允许边缘裁切；远端使用端墙收口柜列而不是纯空墙。
- 原因：用户当前的负面反馈集中在“空”“丑”“像仓库”，本质是设备没有成为视觉主角。
- 结果：
  - 中央通道仅作为引导线，而不是大面积亮地板。
  - 左右近景都需要有柜体进入画面前缘。

### Decision 2: 柜列组织改为前、中、后分层，而非均匀撒点

- 方案：将 32 台设备重组为 7 组视觉柜列：
  - 入口左前主柜群
  - 入口右前主柜群
  - 左中连续柜列
  - 右中连续柜列
  - 左后控制柜列
  - 右后控制柜列
  - 尾端收口柜列
- 原因：即使设备数量不变，只要构图平均，画面就会显得空且弱。
- 结果：
  - 用户从任意方向旋转都能看到设备。
  - 画面存在明确的近中远节奏，而不是“房间中心 + 边缘机柜”。

### Decision 3: 房间几何从“整块地板”改为“中轴通道 + 设备平台”

- 方案：地面分为三段：
  - 中间窄通道
  - 左设备平台
  - 右设备平台
  - 平台与通道之间保留压条/沟槽
- 原因：当前过宽地板直接放大了空旷感。
- 结果：
  - 设备有清晰站位边界。
  - 通道更像控制室通路，而不是展厅大走廊。

### Decision 4: 默认镜头采用偏轴低机位，直接服务设备可见性

- 方案：
  - 默认镜头略偏轴，不站在空间中线
  - 视线优先指向一侧近景主柜与中段纵深
  - FOV 控制在更克制的范围
  - 开场只做短距离平滑推进
- 原因：当前镜头过于中性，导致空间被拉空。
- 结果：
  - 第一眼聚焦设备
  - 地板和空墙不会先于设备进入认知

### Decision 5: 设备家族差异以轮廓和面板结构表达，不再依赖门板换色

- 方案：继续复用现有 `visualFamily` 数据，但在渲染层强化：
  - 柜体宽高比例差异
  - 抽屉/门板/格栅/仪表布局差异
  - 近景和中景使用不同程度的细节密度
- 原因：当前“同模换色”导致设备看起来像复制出来的长方体。
- 结果：
  - 用户可以一眼分出开关柜、控制柜、功率柜、网络柜等家族。

### Decision 6: 场景 UI 叠层从后台卡片改为场景内磨砂铭牌

- 方案：
  - 保留右侧外层 light shell 容器
  - 但标题板改成深色磨砂牌、细描边、小体积，弱化后台卡片感
  - 数量 badge 内嵌在标题板内而非二次抢戏
- 原因：当前标题板与 3D 场景视觉语言割裂。
- 结果：
  - 页面整体更完整，不像“3D 场景上又压了一张后台卡片”。

## Risks / Trade-offs

- [Risk] 过度压缩通道会让相机交互变得局促  
  → Mitigation: 保持室内边界约束不变，但把“视觉宽度”与“几何可操作宽度”分离，必要时用平台台阶而不是简单缩放所有坐标。

- [Risk] 程序化几何仍可能不够写实  
  → Mitigation: 优先提升轮廓、构图、材质分层和近景细节密度，这些对观感提升最大。

- [Risk] 新视觉方向与现有 `dashboard-scene-daylight-visuals` 主 spec 产生冲突  
  → Mitigation: 在本 change 中显式修改该 capability，把要求更新为“现代室内控制室视觉”，不再沿用 daylight 审美。

- [Risk] 场景 UI 叠层改动会影响 light shell 一致性  
  → Mitigation: 只收敛右侧仿真容器内部叠层风格，外层整体仍保持白色 dashboard shell。

## Migration Plan

1. 先重构 `ROOM_LAYOUT` 和地面/平台几何，解决地板过宽与边缘留白问题。
2. 再重构 `buildScenePlacements`，把 32 台设备改为前、中、后、尾端的分层柜列。
3. 调整默认镜头、intro 动画和室内视角约束，使默认视图直接服务设备可见性。
4. 强化柜体家族渲染差异和近景细节。
5. 调整 `main-right` 容器内标题板与场景叠层样式。
6. 进行本机视觉验收和交互回归，确认 1:1 设备数量未破坏。
7. 更新 README，记录新的视觉目标和布局原则。

## Acceptance Heuristics

- 默认开场第一眼看到的是近景设备，不是大块地板。
- 可视地板面积不应成为主视觉主体。
- 四个主要方向旋转后都能看到设备，且不会出现整段空墙主导画面。
- 不同 `visualFamily` 可以通过轮廓和面板结构区分。
- 标题板气质与场景一致，不再像独立后台卡片。
- 前端演示设备数与数据库设备数仍保持 32:32。
