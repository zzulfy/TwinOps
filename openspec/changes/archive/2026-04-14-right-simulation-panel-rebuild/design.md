## Context

本次变更已锁定范围：只改 Dashboard 右侧仿真画面，不重构整页信息架构。  
当前代码路径中，右侧仿真交互主要由 `useDashboardScene` + `DashboardPage` 组合完成，设备详情由 `SimulationDeviceDialog` 渲染。  
既有问题集中在交互命中、展示时机、以及仿真对象与数据库映射一致性治理。

## Goals / Non-Goals

**Goals**

- 只在点击设备本体时显示设备信息。
- 设备信息以页面中央对话框展示，未点击时不出现冗余对话元素。
- 所有业务设备对象均可点击并能命中真实设备数据。
- 仿真设备对象与 MySQL 设备数据建立 1:1 严格对齐机制。
- 状态颜色为透明红黄绿，且不影响模型可见性。

**Non-Goals**

- 不改左侧仪表卡片、顶部导航、其它页面路由。
- 不在本变更中实现完整 IDC 三维漫游系统（如机柜内部镜头切换）全功能版。
- 不修改用户本机网络环境或系统配置。

## Decisions

### Decision 1: 重构范围限定为右侧仿真画布容器

- 只调整右侧仿真区 DOM、场景交互、弹窗样式与映射逻辑。
- 保持 Dashboard 其它业务组件和接口调用节奏不变。

原因：
- 降低变更风险，避免一次性重构整页导致回归面过大。

### Decision 2: 设备信息入口统一为“设备点击 -> 中央弹窗”

- 交互状态仅依赖 `selectedDeviceCode`。
- 关闭路径保留：关闭按钮、遮罩点击、`Esc`。
- 未选中设备时不渲染任何设备详情弹窗。

原因：
- 避免“悬浮在设备上的小对话框”遮挡模型且交互抖动。

### Decision 3: 仿真对象分层管理（可交互 vs 装饰）

```text
[Scene Objects]
  ├─ Interactive Device Objects (必须映射数据库设备)
  └─ Decorative Objects (不参与设备点击和状态渲染)
```

- 仅 Interactive 对象参与命中、状态色覆盖、详情弹窗。
- Decorative 对象保持纯视觉展示。

原因：
- 避免“点到无业务意义对象”却无反馈的体验问题。

### Decision 4: 1:1 一致性以“仿真设备对象清单”为主约束，数据库侧对齐

- 维护仿真可交互设备对象清单（来自模型对象名与右侧画面配置）。
- 后端按该清单生成/校验数据库设备集合。
- 若不一致，优先通过 seed/迁移脚本修正数据库（补齐或删除），而非前端临时兜底显示。

原因：
- 用户明确要求最终表现必须 1:1，不接受长期“显示层兼容偏差”。

### Decision 5: 状态渲染由前端负责，数据获取由后端负责

- 后端仅返回设备状态与指标数据（不返回 UI 色值）。
- 前端按规则映射透明色：`error` 红、`warning/warn` 黄、`normal` 绿。

原因：
- 保持职责边界清晰，减少耦合。

## Risks / Trade-offs

- [Risk] 模型对象命名不稳定导致映射断裂  
  -> Mitigation: 引入对象名归一化与映射白名单测试。

- [Risk] 强制 1:1 可能导致历史设备数据被清理  
  -> Mitigation: 通过脚本先导出差异报告，再执行补齐/删除。

- [Risk] 中央弹窗与右侧画面叠加后影响局部可视  
  -> Mitigation: 控制弹窗尺寸并提供快速关闭。

## Migration Plan

1. 从 `frontend/example/img1`、`img2` 与现有场景模型抽取“可交互设备对象清单”。
2. 落地数据库对齐策略（seed/迁移脚本），确保对象与设备数据 1:1。
3. 重构右侧仿真区交互：统一点击命中、中央弹窗、状态透明色渲染。
4. 清理无意义提示元素与未点击状态下的弹窗渲染。
5. 执行单元测试、集成测试（localhost）、回归测试。
6. 更新 README（记录本次变更范围、交互规范、测试结果）。
