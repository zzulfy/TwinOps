## Context

当前告警能力已有统一后端接口（`fetchAlarmList` 对应 `/api/alarms`），仪表盘告警面板也已使用该接口。但页脚“设备告警”弹窗仍由 `LayoutFooter.vue` 内部硬编码列表驱动，导致后端离线时仍展示伪数据，破坏告警可信度。此次变更仅涉及前端数据源一致性，不改动后端 API。

## Goals / Non-Goals

**Goals:**
- 页脚告警弹窗与其他告警入口统一使用后端告警接口作为唯一数据源。
- 移除页脚告警 mock 数据与相关派生映射逻辑。
- 后端不可用时以可感知的错误/空态反馈替代伪数据展示。

**Non-Goals:**
- 不调整后端告警接口协议、分页语义或状态机。
- 不新增新的告警业务流程（如批量操作、新增筛选维度）。
- 不改动与数据源一致性无关的 UI 风格与动画。

## Decisions

1. **页脚数据源统一到 `fetchAlarmList`（默认 `new` 状态）**
   - 方案：在 `LayoutFooter` 打开弹窗时触发告警拉取，并将结果透传给 `AlarmDeviceList`。
   - 备选：继续使用本地常量作为降级兜底。
   - 选择理由：告警信息属于运维关键数据，必须以服务端真实状态为准，禁止伪造兜底。

2. **错误处理采用“显式错误 + 空态”而非静默回退**
   - 方案：接口失败时在弹窗内显示错误文案；无数据时显示空态。
   - 备选：失败时继续显示上次本地缓存或 mock。
   - 选择理由：静默回退会误导用户认为系统存在告警，增加误判风险。

3. **复用既有 `AlarmDeviceList` 展示契约，最小化组件侵入**
   - 方案：`AlarmDeviceList` 增加加载/错误/空态渲染支持；保留设备条目渲染结构。
   - 备选：重写新的弹窗组件。
   - 选择理由：复用可降低改动面与回归风险，并保持交互一致。

## Risks / Trade-offs

- **[Risk] 后端离线时弹窗不再有“可见数据”** → **Mitigation**：提供明确错误文案和空态，避免用户误以为无异常。
- **[Risk] 打开弹窗即请求可能增加接口调用频率** → **Mitigation**：按“弹窗打开时加载”策略并控制默认 `limit`，必要时可后续加短时缓存。
- **[Risk] 告警字段映射差异造成展示不完整** → **Mitigation**：严格对齐 `AlarmListItem` 字段，缺失值使用可读占位文案。

## Migration Plan

1. 移除 `LayoutFooter.vue` 中硬编码告警数组与 `computed` 映射。
2. 在 `LayoutFooter.vue` 中接入 `fetchAlarmList`，在弹窗打开时加载并维护 loading/error 状态。
3. 更新 `AlarmDeviceList.vue`，支持 `loading`、`errorMessage`、空态展示。
4. 联调验证：后端启动时展示真实告警；后端关闭时展示错误/空态且不出现 mock 项。
5. 回滚策略：若出现严重回归，回滚该 change 对应前端改动，恢复上一版本行为（不建议长期保留）。
