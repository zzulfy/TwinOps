## Context

当前仓库将前端运行资产分布在根目录（如 src、public、index.html、Vite 配置、测试脚本），目录边界与职责不清。该变更目标是在不改变前端功能行为的前提下，将前端资产统一收敛到 frontend 子目录，并确保开发、构建、预览及静态资源加载稳定可用。

约束条件：
- 现有 Vue 3 + Vite + Three.js 能力必须保持可运行。
- 资源路径（models、textures、fonts、draco）迁移后不能失效。
- 现有命令与 CI 可通过最小改动完成切换。

## Goals / Non-Goals

**Goals:**
- 建立明确的前端目录边界：frontend 作为前端工作区根。
- 保持迁移前后运行行为一致（dev/build/preview）。
- 形成可回滚的迁移步骤，降低中断风险。
- 统一文档与脚本入口，减少团队使用歧义。

**Non-Goals:**
- 不引入新的业务功能。
- 不重构组件逻辑或 Three.js 渲染架构。
- 不进行 UI 风格改版。

## Decisions

1. 采用“一次性目录迁移 + 配置改写”的策略
- 方案：将前端相关文件整体迁入 frontend，并更新路径相关配置。
- 备选：逐目录分批迁移。
- 选择理由：一次性迁移更容易保持路径一致性并减少长期双路径兼容成本。

2. 以前端工作区为执行根，保留仓库级入口适配
- 方案：主要开发命令在 frontend 下执行，仓库根可保留代理命令或文档指引。
- 备选：继续在仓库根执行并通过复杂路径重定向。
- 选择理由：降低配置复杂度，便于未来拓展 monorepo 结构。

3. 资源路径采用显式校验清单
- 方案：对模型、纹理、字体、draco 解码器路径建立迁移校验项。
- 备选：仅依赖人工打开页面目测。
- 选择理由：可重复验证，减少遗漏。

4. 测试脚本与检查脚本跟随前端归档
- 方案：将前端相关 test/check 脚本迁入 frontend，并统一命令入口。
- 备选：脚本留在仓库根目录。
- 选择理由：避免路径耦合，提升前端资产内聚性。

## Risks / Trade-offs

- [资源相对路径失效] → 在迁移后执行静态资源清单检查，并覆盖关键页面冒烟验证。
- [CI 工作目录不一致导致构建失败] → 同步更新 CI 配置并新增迁移后构建校验步骤。
- [团队本地脚本使用习惯被破坏] → 提供迁移说明与兼容命令过渡期。
- [一次性迁移改动面较大] → 采用分阶段提交（目录移动、配置修复、验证）并保留回滚点。
