## Context

当前 Dashboard 右侧仿真区已经具备“点击设备本体 -> 中央详情弹窗”的交互，但设备 UI 对象定义与后端数据字段仍存在边界混合风险：后端返回中夹带了部分前端渲染语义，导致前端改 UI 时需要联动后端字段。该变更需要把“UI 结构定义”与“运行数据”完全解耦，前端固定渲染设备对象，后端仅提供实时业务数据。

约束：
- 不能破坏现有点击交互和设备状态色渲染（透明红/黄/绿）。
- 必须保持仿真对象与数据库设备 1:1 对齐校验能力。
- 不引入新的运行时依赖，保持当前前后端技术栈。

## Goals / Non-Goals

**Goals:**
- 在前端建立稳定的仿真设备 UI 配置源（对象、标签、点击区域、deviceCode 映射）。
- 后端设备接口只返回业务数据字段（状态、CPU、温度、湿度、功率、告警、位置等），不返回 UI 布局语义。
- 在前端渲染层实现配置集合与后端数据集合 1:1 校验，发现缺失/多余时可观测且可诊断。
- 保持现有中央弹窗交互链路和视觉状态分层。

**Non-Goals:**
- 不重做左侧业务面板与全站导航。
- 不引入新的三维引擎或替换现有场景渲染框架。
- 不在本次变更中扩展新的业务指标来源（仅整理职责边界）。

## Decisions

### Decision 1: 仿真 UI 对象定义前置到前端静态配置

- 方案：新增前端配置模块（例如 `simulationDeviceUiConfig`），包含设备对象标识、固定标签、业务编码映射、可点击属性。
- 原因：UI 结构属于前端渲染关注点，放在前端可独立演进并避免 API 频繁变更。
- 备选：
  - 继续由后端下发 UI 对象：会放大跨端耦合，否决。
  - 运行时从模型文件动态反推全部映射：稳定性差、调试成本高，否决。

### Decision 2: 后端接口收敛为“数据合同”

- 方案：设备相关 API 输出仅保留业务数据字段，不包含 UI 结构字段（如前端布局坐标、渲染层标签策略等）。
- 原因：后端职责是数据采集、聚合和状态输出；UI 决策应由前端完成。
- 备选：
  - 返回“数据+UI 混合 DTO”：短期省事但长期维护成本高，否决。

### Decision 3: 前端做集合一致性校验并阻断静默错误

- 方案：加载后端设备数据后，按 `deviceCode` 与前端 UI 配置集合比对；若不一致，输出明确诊断信息并进入受控降级展示（仍可查看已有设备）。
- 原因：可在第一时间暴露“UI 有设备但无数据”或“有数据但无 UI”问题，避免上线后隐性错漏。
- 备选：
  - 仅依赖后端一致性校验：前端无感知，问题定位慢，否决。

### Decision 4: 保持现有交互链路不变，仅替换数据来源

- 方案：继续复用点击命中、中央弹窗、状态色渲染逻辑，只改设备集合来源与字段映射层。
- 原因：降低回归风险，便于通过现有 smoke/regression 套件验证。

## Risks / Trade-offs

- [Risk] 前端配置与后端数据在新增/下线设备时不同步  
  → Mitigation: 增加启动校验日志、测试用例和 CI 阻断规则，要求集合一致。

- [Risk] 移除后端 UI 字段后，历史前端代码仍读取旧字段导致空值  
  → Mitigation: 增加 TypeScript 类型收敛与契约测试，逐点替换旧字段引用。

- [Risk] 配置文件规模增大后可维护性下降  
  → Mitigation: 按区域分组配置并规范命名，提供注释与校验脚本。

## Migration Plan

1. 定义前端 UI 配置数据结构并迁移现有设备映射。
2. 调整前端场景绑定层，改为“UI 配置 + 后端业务数据”合并渲染。
3. 收敛后端 DTO 字段，移除 UI 语义字段并更新控制器/服务测试。
4. 增加前端/后端一致性测试与集成回归验证。
5. 更新 README 记录职责边界和测试命令。

回滚策略：
- 若上线验证失败，可暂时保留后端旧字段并在前端开启兼容读取开关，之后再二次清理。

## Open Questions

- UI 配置中“固定标签文本”是否全部来自业务命名规范，还是允许前端别名展示？
- 后端 `location` 字段是否保留为业务数据（建议保留），由前端决定展示格式。
