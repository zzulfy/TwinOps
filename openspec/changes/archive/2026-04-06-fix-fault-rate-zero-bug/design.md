## Context

当前 Dashboard 故障率曲线在分钟时间轴和文案语义上已完成规范化，但核心值计算仍使用与设备故障占比不直接等价的替代信号，导致数据库中存在 `error` 设备时仍可能展示为 0。业务侧定义已经明确：故障率必须来自“`error` 设备数 / 全部硬件设备数”，因此需要将计算来源切换到设备状态聚合并与现有分钟刷新链路保持兼容。

## Goals / Non-Goals

**Goals:**
- 将故障率计算统一为设备状态占比：`errorCount / totalDeviceCount * 100`。
- 在保持分钟级趋势接口与 UI 交互不变的前提下，修复故障率误报为 0 的问题。
- 明确边界行为（无设备、无 error）并补充可回归的测试断言。

**Non-Goals:**
- 不重构 AI 预测算法与趋势外观风格。
- 不修改非故障率相关模块（鉴权、告警流转、分析中心）。
- 不新增独立故障率 API，仅在现有链路内修正计算口径。

## Decisions

1. **以设备状态作为故障率唯一数据源**
   - 后端在汇总故障率时读取设备状态字段，按 `status = error` 统计分子，设备总数为分母。
   - 理由：与数据库业务定义一致，可直接解释给运维人员。
   - 备选方案（沿用 telemetry/cpu 派生）被拒绝：无法保证与“error 设备占比”语义一致。

2. **保留分钟级时间轴展示机制**
   - 故障率时间点仍按分钟窗口输出，标签保持 `HH:mm`；但每个点值由状态占比计算得出。
   - 理由：满足现有 UI 监控节奏与交互习惯，修复问题同时避免不必要改动。

3. **边界值显式定义并测试化**
   - 无设备时故障率定义为 0，存在设备且有 error 时故障率必须大于 0。
   - 通过 service/controller 测试覆盖关键场景，避免回归再次出现“恒 0”。

## Risks / Trade-offs

- **[Risk] 设备状态更新延迟会导致短时故障率滞后** → **Mitigation**: 保留分钟刷新并在文档中明确故障率来源是设备状态快照。  
- **[Risk] 历史时间点若无状态快照可能出现平台期** → **Mitigation**: 延续当前序列补齐策略，保证曲线连续可读。  
- **[Trade-off] 从 telemetry 派生值切换为状态占比后，曲线波动可能变小** → **Mitigation**: 这是符合业务定义的预期变化，非精度退化。  

## Migration Plan

1. 更新后端故障率聚合逻辑并保持接口结构兼容。  
2. 更新前端故障率模块文案/说明（如需）以强调占比语义。  
3. 补齐测试与 README 描述后统一发布。  
4. 若出现异常，可回滚到上一版本并保留现有 API 契约不变。
