## Context

当前系统已具备设备数据与三维模型基础，但存在两个核心缺口：

1. 三维场景没有设备身份可视化与点击详情交互，用户无法直接从仿真视图定位到真实设备数据。  
2. 仿真设备与数据库设备缺少统一主键约束与自动修复机制，存在数据漂移风险。

基于当前仓库现状：

- `frontend/public/models/devices.glb` 内存在 51 个有效设备对象名（剔除管道/辅助对象后），可作为仿真设备集合基础。
- `backend/sql/002_seed_devices.sql` 已维护 `DEV001`~`DEV051` 及 `label_key`，具备建立映射主键的条件。
- `label_key` 已在数据库实体层存在，但前端设备 API 契约尚未显式暴露该字段用于仿真绑定。

## Goals / Non-Goals

**Goals**

- 为 51 台仿真设备提供统一、稳定、可见的标签展示（`deviceCode + name`）。
- 支持设备点击后在设备上方弹出悬浮对话框，展示设备详情。
- 建立仿真设备集合与数据库设备集合的 1:1 一致性检查与修复机制。
- 明确并固化不一致修复顺序：先删除数据库冗余设备，再补齐数据库缺失设备。

**Non-Goals**

- 不重建 Three.js 场景结构或替换模型格式。
- 不改造现有业务模块（告警、分析、鉴权）核心逻辑。
- 不将本次能力扩展为生产级多集群主数据治理平台。

## Decisions

### Decision 1: `label_key` 作为仿真-数据库绑定主键

- 前端模型设备对象名与后端 `devices.label_key` 做严格匹配。
- `deviceCode` 用于展示与业务接口调用，`label_key` 用于绑定定位。

原因：
- `deviceCode` 是业务编码，`label_key` 才与仿真对象命名天然对应；
- 能避免“同名设备文案变化导致映射失效”的问题。

### Decision 2: 设备标签与点击对话框都在 Dashboard 三维视图完成

- 常驻标签层：展示每台设备最小识别信息（`deviceCode + name`）。
- 点击交互层：点击设备后显示悬浮对话框，内容复用当前设备详情字段（状态、类型、位置、遥测、告警摘要）。
- 对话框跟随设备锚点，点击空白区域关闭。

原因：
- 与用户使用路径一致，减少“切页查设备”的跳转成本；
- 最大化复用现有 `DeviceData` 字段，减少额外接口复杂度。

### Decision 3: 一致性修复采用“删除优先，再补齐”的确定性策略

修复算法（同一事务窗口内执行）：

1. 计算集合差：
   - `extraInDb = dbLabelKeys - simulationLabelKeys`
   - `missingInDb = simulationLabelKeys - dbLabelKeys`
2. 先删除 `extraInDb` 设备及其关联 metrics/alarms。
3. 再插入 `missingInDb` 设备主数据，并按数据集生成策略补齐 telemetry/alarms。
4. 复检设备总量、孤儿数据、可分析目标数量。

原因：
- 先删除可快速消除“脏设备”对统计和分析的污染；
- 再补齐可恢复完整覆盖。

### Decision 4: 一致性校验结果对前端可见

- 若校验/修复失败，前端场景显示明确错误提示并暂停交互绑定，避免用户看到错误映射结果。
- 若修复成功，前端继续正常渲染和交互。

原因：
- 强一致需求优先于“勉强可用”的弱一致展示。

## Risks / Trade-offs

- [Risk] 模型命名规则变化会导致映射失效  
  -> Mitigation: 抽离统一过滤与命名规范，新增单元测试校验 51 设备清单稳定性。

- [Risk] 自动修复涉及删除设备，存在误删风险  
  -> Mitigation: 仅允许删除不在仿真白名单中的设备，并记录结构化日志与修复摘要。

- [Risk] 51 个标签和实时锚点更新可能增加渲染开销  
  -> Mitigation: 标签层采用轻量化更新策略，仅在相机/场景变化时刷新坐标。

- [Risk] 数据集生成脚本缺失导致“补齐”不可复现  
  -> Mitigation: 本变更纳入脚本恢复/补齐任务，并要求在 CI 或本地验证中执行一致性检查。

## Migration Plan

1. 扩展设备数据契约，确保前端能拿到仿真绑定所需键（`labelKey`）。  
2. 实现前端标签层与点击悬浮对话框，并完成模型对象到设备数据映射。  
3. 增加后端一致性检查与修复流程（删除优先、补齐其次）。  
4. 补齐/恢复数据集驱动生成脚本，并更新 SQL 校验脚本。  
5. 执行单元测试、集成测试、回归测试并记录证据。  
6. 更新 README 中的仿真设备映射与一致性运维说明。
