## Context

TwinOps 前端当前在可观测展示与交互实时性上存在复合问题：部分告警展示仍可能出现测试/占位数据、告警卡片在深色主题下存在低对比度组合、分析报告卡片标题信息冗余、核心数据更新依赖用户手动刷新浏览器、自动刷新机制与性能策略缺少统一约束。该变更覆盖 Dashboard、Alarm、Analysis 等多个页面与共享数据获取逻辑，属于跨模块前端体验与性能治理。

## Goals / Non-Goals

**Goals:**
- 前端关键业务展示仅使用后端真实数据，不再落入 mock 展示分支。
- 告警卡片在深色主题下具备稳定可读性（文本与背景高对比）。
- 分析中心卡片标题统一为“编号 + 生成时间”主视觉，移除固定 `AGGREGATED` 文案。
- 建立自动刷新基线（定时刷新 + 页面可见性控制 + 手动刷新兼容）。
- 在自动刷新引入后控制请求频率与渲染成本，避免性能回退。

**Non-Goals:**
- 不新增后端 API，不改变后端分析任务语义。
- 不重做整体 UI 风格，仅修复可读性与信息层级问题。
- 不引入重型实时通道（如 WebSocket）替代现有 HTTP 拉取。

## Decisions

- **Decision 1: 统一真实数据入口并移除 mock fallback 展示**  
  对告警/分析展示组件统一要求“请求失败显示错误态或空态”，禁止继续渲染本地测试数据，保证运维判断基于真实状态。

- **Decision 2: 使用设计 token 修复告警卡片对比度**  
  优先复用 `design-tokens.css` 的语义色，避免组件内硬编码颜色；通过前景色与背景色组合修复深色对深色不可读问题。

- **Decision 3: 分析卡片主标题去语义化标签，强化时间与编号**  
  去除固定 `AGGREGATED` 标题，卡面首屏只保留报告编号与生成时间两类关键信息，并提升字号层级以支持快速扫描。

- **Decision 4: 自动刷新采用“定时轮询 + 可见性门控 + 手动补充”**  
  在页面可见时按固定间隔刷新，页面隐藏或失焦时降频/暂停；保留手动刷新作为应急入口，避免完全依赖被动等待。

- **Decision 5: 以低扰动性能优化为主线**  
  针对自动刷新场景引入请求去重、并发保护、最小必要重渲染与大模块延迟加载复用，降低网络与渲染抖动。

## Risks / Trade-offs

- **[Risk]** 自动刷新导致请求增多并放大后端负载 → **Mitigation**: 设置合理刷新间隔，页面隐藏时暂停，避免并发重复请求。
- **[Risk]** 去除 mock fallback 后，后端异常时页面“看起来数据更少” → **Mitigation**: 提供明确 error/empty 状态文案，避免误判为正常数据。
- **[Risk]** 色彩调整影响既有视觉习惯 → **Mitigation**: 仅调整低对比组件，保持其他视觉 token 不变。
- **[Trade-off]** 更高实时性与更低资源消耗存在张力 → **Mitigation**: 提供统一 cadence 策略与可扩展配置，按页面重要性分层刷新。

## Migration Plan

1. 先落地 OpenSpec requirements（自动刷新、真实数据、卡片可读性与分析卡面标题）。
2. 实施前端组件与 composable 改造，保持 API 契约不变。
3. 增加并执行前端 type-check/build 与回归验证（自动刷新行为、告警可读性、分析卡片标题）。
4. 更新 README/前端文档中的刷新与性能约束说明。

## Open Questions

- 自动刷新 cadence 是否按页面分级（如 Dashboard 更快、Analysis 稍慢）需要在实现阶段结合体验进一步细化。
- 页面不可见时采用“完全暂停”还是“降频轮询”可在性能验证后择优确定。
