## Context

当前前端已完成 React 化与模块化拆分，但不同页面在“模块入口位置、信息展示区域、宽度占比、交互反馈”上仍不一致。用户在 Dashboard、Devices、Analysis 间切换时需要重新理解布局，无法形成稳定操作心智。  
本次变更在不调整业务逻辑与 API 数据流的前提下，统一为 ChatGPT 风格的 two-pane 信息架构：左侧窄列导航、右侧宽列内容。

## Goals / Non-Goals

**Goals:**
- 建立全局 `AppShell`，统一页面的左导航右内容结构。
- 左侧承载模块标签与入口，右侧承载当前模块详细内容。
- 明确左右列比例与响应式规则，保证桌面端优先、窄屏可退化。
- 保持现有业务行为（鉴权、路由、请求、自动刷新、告警流转）不变。

**Non-Goals:**
- 不修改 backend API、DTO、鉴权协议。
- 不重写模块业务组件内部逻辑。
- 不引入新状态管理库或新 UI 组件库。

## Decisions

### Decision 1: 引入统一 AppShell 布局层
- **选择**：新增 `AppShell` 作为页面结构壳层，所有主模块页面在该壳层内渲染。
- **原因**：避免每个页面重复实现侧栏与内容区域，实现一致的信息架构与交互基线。
- **备选方案**：在每个页面单独重构 two-pane。  
  **未采用原因**：重复成本高，后续演进容易失配。

### Decision 2: 左窄右宽采用固定边界 + 弹性比例
- **选择**：左列使用 `clamp()` 控制（例如 220~320px），右列 `1fr` 占据剩余空间。
- **原因**：可在不同分辨率下保持“左窄右宽”视觉稳定，并避免超宽/超窄失真。
- **备选方案**：纯百分比（如 20/80）。  
  **未采用原因**：在超宽或窄屏下体验不稳定。

### Decision 3: 模块入口统一收敛到左列
- **选择**：Dashboard、Devices、Analysis 等主模块入口只保留在左列导航。
- **原因**：避免入口散落在内容区，降低认知负担。
- **备选方案**：保留原页面内重复入口。  
  **未采用原因**：入口重复且状态不一致，破坏统一体验。

### Decision 4: 业务逻辑冻结，布局层改造
- **选择**：仅调整结构与样式，业务组件尽量原位复用。
- **原因**：降低回归风险，确保“布局升级而非业务重构”。
- **备选方案**：同步重构业务状态与组件边界。  
  **未采用原因**：风险过大，不符合本次范围。

## Risks / Trade-offs

- **[Risk]** 旧页面样式与新壳层冲突（定位、z-index、overflow）  
  **→ Mitigation**：新增 `appshell-*` 命名空间样式，限制影响面并分阶段替换。

- **[Risk]** 左列收窄后标签信息密度过高  
  **→ Mitigation**：提供 tooltip / 文本截断策略，并保留关键图标与选中态。

- **[Risk]** 响应式断点处理不当导致小屏可用性下降  
  **→ Mitigation**：定义断点下折叠策略（左列可折叠/抽屉化），并补 smoke 场景验证。

- **[Trade-off]** 初期会产生一层壳组件抽象  
  **→ Mitigation**：通过统一布局与导航协议换取长期维护一致性。

## Migration Plan

1. 新增 `AppShell` 与 two-pane 样式 Token，先接入 Analysis 页面验证。
2. 将 Devices 页面接入 `AppShell`，合并散落入口到左列导航。
3. 将 Dashboard 页面接入 `AppShell`，处理原浮层布局到右侧主区映射。
4. 统一导航状态与路由映射，移除内容区重复入口按钮。
5. 回归验证：type-check、build、smoke（shell/analysis/alarm）与 backend test。

Rollback：
- 保留旧页面布局分支点，按页面粒度回退 `AppShell` 接入，确保可快速恢复。
