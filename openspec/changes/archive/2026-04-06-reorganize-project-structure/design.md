## Context

TwinOps 当前代码库虽已完成 frontend 子工作区迁移，但 `frontend/` 与仓库根目录仍保留较多历史脚本、临时日志与一次性验证文件，导致目录语义弱化、定位成本高、改动范围难以评估。本次设计聚焦“结构治理（structure governance）”而非功能改造，要求在保持业务逻辑不变的前提下，完成文件归档与路径收敛。

## Goals / Non-Goals

**Goals:**
- 建立明确的 repository file taxonomy，区分源码、配置、脚本、报告、临时文件与数据样例。
- 收敛 `frontend/` 与 root 的散落文件，降低目录噪音与误改风险。
- 保持现有功能行为、API contract 与页面逻辑不变。
- 保证迁移后命令链路可执行，文档与实际目录一致。

**Non-Goals:**
- 不新增或修改任何业务功能。
- 不进行 Vue/React 业务层重构。
- 不更换构建工具链或引入新的框架。

## Decisions

1. 采用“分类归档 + 路径映射修复”的整理策略
- 方案：先定义目标目录分类，再按映射执行文件迁移与引用更新。
- 备选：仅删除临时文件，不做结构化归档。
- 选择理由：仅删除无法解决后续持续沉积问题，分类归档可形成长期可执行约束。

2. 对 frontend 目录执行最小根目录原则（minimal root policy）
- 方案：`frontend/` 根目录只保留运行与构建入口文件；检查脚本、调试脚本、验证产物迁移到子目录。
- 备选：维持现状，通过命名约定管理。
- 选择理由：目录层级治理优于命名约定，减少认知负担。

3. 根目录保持“项目入口层”，非入口文件下沉到域目录
- 方案：root 仅保留 README、顶层业务目录与必要入口配置；数据样例与日志归档到专用目录。
- 备选：继续在 root 放置跨模块辅助文件。
- 选择理由：降低根目录噪音，提高新成员 onboarding 效率。

4. 迁移时禁止业务语义变更
- 方案：仅做 path relocation 与 import/script path 修复，不改业务逻辑与接口契约。
- 备选：整理结构时顺带优化业务代码。
- 选择理由：避免变更耦合，降低回归风险。

## Risks / Trade-offs

- [路径迁移导致脚本失效] → 维护迁移映射表并逐项修复 npm scripts/文档命令。
- [历史临时文件被误删] → 先分类归档到 `tmp/` 或 `reports/`，确认后再清理。
- [CI 或本地自动化依赖旧路径] → 对受影响路径建立兼容说明，并同步更新调用点。
- [改动面广但收益短期不显著] → 通过规范化目录与可验证验收标准，形成长期维护收益。

## Migration Plan

1. 盘点 root 与 `frontend/` 散落文件，生成“旧路径 -> 新路径”映射。
2. 创建目标目录并迁移文件（脚本、日志、报告、样例数据、临时文件）。
3. 修复引用路径（npm scripts、文档命令、可能存在的自动化脚本）。
4. 运行既有构建与检查命令，确认功能行为不变。
5. 更新 README 的目录结构与使用说明。
